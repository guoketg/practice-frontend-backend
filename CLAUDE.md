# CLAUDE.md — AI 编程助手通用工作规范

本文件为 AI 编程助手（如 CodeBuddy、Claude 等）在此项目中的工作提供指导规范。

---

## 核心规则：直接回复优先（Reply-First）

**禁止在 `gudiance/` 目录下创建或编辑 `.md` 引导文件**，除非用户**明确要求生成文档**。

本项目的目标是快速开发和迭代，而非学术教学。AI 应当直接在聊天中输出技术指导，让用户自行决定是否应用到代码中。

### 何时直接对话回答（默认行为）

以下所有情况均在聊天中**直接回复**，不生成 `.md` 文件：

- **功能实现指导**："如何添加编辑功能"、"怎么实现注销"、"添加一个登录页面"、"重构这个组件"
- **概念讲解**："useEffect 钩子怎么用"、"React 的 useState 是什么"
- **调试/排错**：导入错误、构建失败、拼写错误、路径问题
- **快速事实性问题**："这个命令是干什么的"、"npm run dev 在哪个端口"
- **简单的一步修复**："为什么我的图片不显示"、"这个 CSS 不管用"

回复格式应遵循：**操作类型 + 文件路径 + 行号 + 修改前/后代码对比**（见下方《AI 代码操作规范》）。

### 何时创建引导文档（`.md` 文件）

**仅在用户明确要求时**才在 `gudiance/` 下创建 `.md` 文件。示例用语：

- "帮我写一份引导文档"
- "生成 .md 文件"
- "把这个写到 gudiance 里"

### 何时直接修改代码

AI **仅在用户明确要求时**才可直接修改源代码文件。示例用语：

- "请直接修改我的代码。" / "直接修改代码" / "帮我应用这个改动。"
- "修复这个文件的 bug。" / "帮我改一下"

### 快速参考表

| 用户意图 | AI 行为 |
|---|---|
| 功能实现 / 概念讲解 / Bug / 排错 / 快速问题 | 在聊天中**直接回复**（默认） |
| 用户明确说"生成文档"/"写 .md" | 在 `gudiance/` 下创建 `.md` 文件 |
| 用户明确说"直接修改"/"帮我改代码" | 直接编辑源代码文件 |
| 用户提到 `.md` 文件名并要求**禁止/不要** | 绝不为该需求创建或编辑 `.md` 文件 |

## AI 代码操作规范：必须精确到文件、行号、操作类型

当 AI 在引导文档中提供代码修改指导时，**必须使用以下格式精确描述每一步操作**，让用户清楚地知道在哪里做什么：

### 操作类型定义

| 操作类型 | 说明 | 示例描述 |
|---|---|---|
| **新建文件** | 创建新文件 | `新建文件 backend/utils.py，写入以下代码` |
| **删除文件** | 删除整个文件 | `删除文件 backend/old_module.py` |
| **编辑文件** | 修改文件中的某段代码 | `编辑文件 backend/main.py 第 114-127 行，将 xxx 改为 yyy` |
| **新增代码** | 在文件中新增代码块 | `在 backend/orm.py 第 80 行之后新增以下代码` |
| **删除代码** | 删除文件中的某段代码 | `删除 backend/main.py 第 45-50 行` |

### 编写规范

每条操作指导必须包含以下三个要素：

1. **操作类型**：新建文件 / 删除文件 / 编辑文件 / 新增代码 / 删除代码
2. **精确位置**：文件路径 + 行号（对于已有文件）
3. **具体内容**：修改前是什么、修改后是什么（附带代码块）

### 正确示例 vs 错误示例

**正确示例：**

> **步骤一：编辑文件 `backend/main.py` 第 7 行**
>
> 在导入语句中添加 `RegisterRequest`：
>
> ```python
> # 修改前
> from orm import Base, DepartmentCreate, Department, EmployeeCreate, Employee, Attendance, AttendanceCreate, LoginResponse, LoginRequest
> 
> # 修改后
> from orm import Base, DepartmentCreate, Department, EmployeeCreate, Employee, Attendance, AttendanceCreate, LoginResponse, LoginRequest, RegisterRequest
> ```

**错误示例（禁止使用）：**

> 在 main.py 里加个 import。
>
> ```python
> from orm import RegisterRequest
> ```

原因：没有说明操作类型、没有指明行号、没有展示修改前后的对比。

## 引导文档编写规范（仅在用户明确要求时使用）

当用户**明确要求**在 `gudiance/` 下创建 `.md` 引导文件时，应遵循以下结构：

1. **标题**：`# 功能名称实现指导`
2. **需求分析**：当前存在什么问题，需要完成什么
3. **前置修复**（如有）：先修复已有的 bug
4. **修改步骤**：按文件分步，每一步包含：
   - 操作类型 + 文件路径 + 行号
   - 修改前代码 vs 修改后代码
   - 修改原因说明
5. **接口响应格式**：完整的成功/失败 JSON 示例
6. **测试方法**：curl 命令或 Swagger UI 测试方式
7. **注意事项**：安全性、性能、边界情况等
8. **代码风格规范**（教学文档必遵守）：
   - **先讲作用，再给代码**：每个代码块之前先用一两句话说明「这段代码/文件在项目中起什么作用、对应后端的什么概念」，不要一上来就贴代码逐行讲。
   - **代码块保持精简**：代码内不堆砌行内注释（尤其禁止逐行 Python 类比注释），保持干净、可直接运行的代码。
   - **讲解后置**：对代码的细节讲解放在代码块**之后**，用文字段落分点说明，不要穿插在代码行内。
   - **文件级代码必须完整**：所有「文件级代码块」必须是完整、可直接保存运行的版本，不要给片段。

⚠️ **再次强调：默认不要在 `gudiance/` 下创建 `.md` 文件。直接对话回复是唯一默认行为。**
