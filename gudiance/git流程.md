改代码前 & 改完后的标准流程

  第一步：开工前检查（5 条命令）

  # 1. 看本地有没有没保存的改动
  git status

  # 2. 看本地和远程的分支追踪关系
  git branch -vv

  # 3. 下载远程最新信息（只下载不改代码）
  git fetch

  # 4. 再次确认本地和远程的关系
  git branch -vv

  # 5. 确定没有分叉或落后太多
  # 如果 branch -vv 显示 behind，那就需要 pull

  第二步：决定怎么合并

  看 git branch -vv 的输出：

  ┌───────────────────┬────────────────┬─────────────────────┐
  │       显示        │     意味着     │        做法         │
  ├───────────────────┼────────────────┼─────────────────────┤
  │ behind 0, ahead 0 │ 本地和远程一致 │ 直接开始改代码      │
  ├───────────────────┼────────────────┼─────────────────────┤
  │ behind 3          │ 远程有更新     │ 需要 pull           │
  ├───────────────────┼────────────────┼─────────────────────┤
  │ ahead 2, behind 1 │ 分叉了         │ 需要处理分叉        │
  ├───────────────────┼────────────────┼─────────────────────┤
  │ ahead 2           │ 本地领先       │ 正常，改完一起 push │
  └───────────────────┴────────────────┴─────────────────────┘

  第三步：同步代码

  # 情况A：本地落后于远程（最常见，也最安全）
  git pull --rebase   # 推荐用 rebase，历史干净，不会多一个合并提交

  # 情况B：分叉了（只要按照标准流程走，分叉很少出现）
  # 如果你确认远程的 1 个提交不重要，用自己的覆盖
  git push --force    # 多人协作尽量别用（因为会把别人的提交覆盖）
  # 或者老老实实合并
  git pull            # 会产生一个合并提交，有冲突就解决

  # 情况C：本地和远程一致
  # 什么都不用做，直接开始改代码

  第四步：改代码 + 提交

  # 改完代码后
  git status                  # 看看改了什么
  git add 改动的文件           # 或 git add .
  git commit -m "做了什么改动" # 提交到本地

  第五步：推送

  # 推送前最后确认一次
  git fetch                   # 看看有没有人在你改代码时推送了新东西

  git branch -vv              # 再看一眼

  # 如果有新东西，先处理：
  git pull --rebase           # 把别人的更新接到你的提交前面(用--rebase主要是可以退回到这个指令之前的状态)

  # 确认无误，推送
  git push

  ---
  流程图总结

  开工前                        改完后
  ─────                        ─────

  git status       确认干净     git status
      ↓                            ↓
  git fetch        拉取更新     git fetch       看有没有人抢先
      ↓                            ↓
  git branch -vv   看清楚关系   git branch -vv   确认没落后
      ↓                            ↓
  git pull --rebase 同步       如果有落后→git pull --rebase
      ↓                            ↓
  开始改代码                    git push         完事

  ▎ 核心就两句：改前先 fetch + pull，改完先 fetch 再 push。多加一步确认，少踩一万个坑。
